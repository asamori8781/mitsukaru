# ミツカル

PC内に散在するファイルを、曖昧なキーワードから探し出すWindows向けデスクトップツールです。

生成AIは「検索キーワードの展開」にのみ使用し、ファイルの走査・検索はすべてローカルで完結します。

- **Phase 0**: ファイル名・パス・拡張子・サイズ・更新日時のみを索引するメタデータ検索(中身は読まない)
- **Phase 1**: 対応ファイル形式の本文を抽出してローカルの全文検索・意味検索(埋め込み)を追加し、
  ファイル名に手がかりがなくても見つけられるようにする(下記「Phase 1: 全文・意味検索」参照)

## 構成

```
config.py     設定(data/config.json)の読み書き・既定値・データ配置パス解決
db.py         SQLiteスキーマ(files/files_fts/file_content/file_chunks)の初期化・バージョンチェック
indexer.py    フォルダ走査・差分スキャン・進捗管理・コンテンツインデックス作成・Phase1予測サイズ算出
ai_client.py  AI API呼び出しの隔離(キーワード展開・接続テスト)。モックモード対応
extractor.py  Phase1: PDF/DOCX/XLSX/PPTX/テキスト系ファイルからの本文抽出
embedder.py   Phase1: ローカル埋め込みモデルのダウンロード・推論・類似度計算
search.py     FTS5(trigram)によるファイル名+本文検索、埋め込みによる意味検索の統合
server.py     FastAPIルーティング・利用ログ記録・静的ファイル配信
main.py       エントリポイント(pywebviewウィンドウ+FastAPIバックグラウンド起動)
static/       フロントエンド(HTML/CSS/JS、フレームワーク不使用)
data/         実行時生成、git管理外(config.json, mitsukaru.db, models/)
logs/         実行時生成、git管理外(api_log.jsonl, usage_log.csv)
```

## 開発環境での起動手順(uvicorn直起動)

pywebviewのウィンドウを使わず、通常のブラウザで動作確認する場合は以下の手順です
(Windows以外の開発機でのフロー確認にも使えます)。

```bash
python -m venv venv
venv\Scripts\activate        # Windowsの場合。macOS/Linuxは source venv/bin/activate
pip install -r requirements.txt
uvicorn server:app --reload --port 8000
```

ブラウザで `http://127.0.0.1:8000/` を開くとUIが表示されます。

実際にpywebviewウィンドウとして起動する場合は以下です(Windows想定)。

```bash
python main.py
```

## 初回設定の流れ

1. `data/config.json` が存在しない場合、起動時にAPI初期設定モーダルが表示されます。
   - ベースURL・APIキー・モデル名・タイムアウト秒(既定20秒)を入力し、「接続テスト」で疎通確認できます。
   - 「スキップ」を選ぶとAPIを使わない**モックモード**で起動し、画面上部に
     「モックモード動作中(API未設定)」と常時表示されます。モックモード・API接続失敗時は、
     入力文を空白で分割した語をそのままキーワードとしてローカル検索にフォールバックします
     (日本語は分かち書きされないため、文中に空白がない場合は文全体が1語として扱われ、
     ヒットしないことがあります。これは仕様上の制約です)。
2. 続けて対象フォルダ選択画面が表示されます。「このPCの全ドライブをスキャン」または
   個別フォルダの追加(フォルダ選択ダイアログ、ブラウザ直起動時はパス直接入力)を選び、
   スキャンを開始します。
3. スキャン完了後、登録件数・DBファイルサイズ・Phase 1予測サイズ(概算)が表示され、
   検索画面に進めます。
4. (任意)全文検索・意味検索を使いたい場合は、設定画面の「コンテンツインデックスを作成・更新」を
   実行してください。初回実行時、意味検索用の埋め込みモデル(約100MB)を自動ダウンロードします。

以降の起動では、アプリ起動時と検索実行時(前回差分スキャンから設定分数以上経過している場合)に
バックグラウンドで自動差分スキャンが走るため、手動メンテナンスは不要です。コンテンツインデックスも
一度作成した後は、差分スキャン完了のたびに変更されたファイルだけが自動で追加処理されます
(設定画面のチェックボックスで無効化できます)。

## PyInstallerでのビルド手順(onedir)

```bash
pip install pyinstaller
pyinstaller --name mitsukaru --onedir --windowed --icon assets/icon.ico --add-data "static;static" main.py
```

- macOS/Linuxでビルドする場合は `--add-data "static:static"`(区切り文字が `:`)にしてください。
- `--icon assets/icon.ico` によりexeとウィンドウのタイトルバー・タスクバーにミツカルのロゴが
  表示されます。開発中の `python main.py` 実行ではWindowsの仕様上Pythonの既定アイコンが
  表示されますが、ビルド後は専用ロゴになります。
- 生成物は `dist/mitsukaru/` 配下に展開されます。`data/` と `logs/` は実行ファイルと同階層に
  自動生成されるため、フォルダごとコピーすればポータブルに動作します。
- **想定サイズ目安**: FastAPI/Starlette/Uvicorn/pywebview/requests一式とPythonランタイムのみなら
  onedir全体でおおよそ数十〜100MB程度ですが、Phase 1の抽出ライブラリ(pypdf/python-docx/
  openpyxl/python-pptx、いずれも純Python)と埋め込み計算用のnumpy/onnxruntime/tokenizersが
  加わるため、実際にはプラス100〜150MB程度(合計200〜300MB程度)を見込んでください
  (pandasは引き続き未使用)。埋め込みモデル本体(約100MB)はビルドに含めず`data/models/`へ
  実行時ダウンロードするため、配布物自体のサイズには含まれません。実測値は開発機での
  ビルド後に確認してください。
- Windowsではpywebviewの既定バックエンド(WebView2)を利用するため、WebView2 Runtime
  (Windows 10/11には標準搭載されていることが多い)が必要です。

## APIキーの取り扱いに関する注意

- `data/config.json` にAPIキーを**平文で保存**します。`.gitignore` で `data/` を
  リポジトリ管理外にしていますが、ファイル自体はローカルディスクに平文で残るため、
  共有PCや会社貸与PCでの取り扱い、バックアップ・持ち出し時には十分注意してください。
- APIへ送信するのはユーザーの検索文と固定の展開指示プロンプトのみです。ファイル名・パス・
  ファイルの内容・スキャン結果は一切送信しません(送信処理は `ai_client.py` に隔離しています)。
- 送受信内容は `logs/api_log.jsonl` にローカル保存されます(透明性ログ、APIキーは記録しません)。
- 別のPCや業務環境へ展開する際は、`data/config.json` を差し替えるだけで接続先を
  切り替えられます(接続情報はコードにハードコードしていません)。

## 検索の仕組みと制約

- 検索対象は**ファイル名・フルパス**に加え、コンテンツインデックス作成済みのファイルは
  **本文**も対象になります(下記「Phase 1: 全文・意味検索」参照)。
- ひらがな/カタカナ・全角/半角の表記ゆれは検索側で自動吸収します
  (例: 「ふくすぽ」で「フクスポ大会管理表.xlsx」がヒットします)。
- OneDrive配下(ファイルオンデマンドのプレースホルダー)もスキャン対象です。
  フォルダのシンボリックリンク・ジャンクションは無限ループ防止のため辿りません。

## Phase 1: 全文・意味検索

Phase 0のファイル名検索に加えて、以下をローカルのみで完結させて実現しています(生成AIの
外部APIには一切送信しません)。

- **対応形式**: `.txt` `.md` `.csv` `.json` 等のテキスト系(そのまま読み込み)、
  `.pdf`(pypdf)、`.docx`(python-docx)、`.xlsx`(openpyxl)、`.pptx`(python-pptx)
- **本文全文検索**: 抽出した本文をファイル名と同じtrigram方式のFTS5で索引し、キーワードが
  本文中に含まれるファイルもヒットするようにします。
- **意味検索(埋め込み)**: 本文をチャンク分割し、ローカルの埋め込みモデル
  (`Xenova/multilingual-e5-small`の量子化ONNX版、多言語対応・約100MB)でベクトル化します。
  検索文自体もベクトル化し、総当たりのコサイン類似度でファイル名・本文どちらにもキーワードが
  一致しなかったファイルを補って提示します。**モックモード時やAI API失敗時でも、この意味検索
  自体は動作します**(検索文の埋め込みにAI APIを使わないため)。
- **有効化**: 設定画面の「コンテンツインデックスを作成・更新」を実行してください。埋め込み
  モデルは初回実行時に自動ダウンロードされます(ネット接続が必要。社内プロキシ等で
  `huggingface.co` に到達できない場合はダウンロードに失敗し、その場合も本文全文検索
  (埋め込み不要)自体は有効になります)。
- **増分更新**: 一度抽出したファイルは、更新日時が変わらない限り再抽出をスキップします。
  埋め込みモデル未ダウンロードのまま抽出した(全文検索のみの縮退運転)ファイルは、
  モデルが使えるようになった後の再実行時に、再抽出せず埋め込みだけ追加で行われます。
- **自動増分更新**: 初回の作成を一度手動で実行した後は、差分スキャン(起動時・検索時の
  自動実行を含む)が完了するたびに、変更・追加されたファイルだけが自動でコンテンツ
  インデックスへ反映されます(設定画面のチェックボックスでOFFにできます)。変更が
  1件も無い場合は埋め込みモデルのロード自体を省略して即終了するため、常時ONでも
  負荷はほぼありません。なお自動実行では埋め込みモデルのダウンロードは試みません
  (オフライン環境で差分スキャンのたびにネットワーク待ちが発生するのを避けるため。
  モデル未取得のままの場合は全文検索のみ反映され、埋め込みは後の手動実行時に
  まとめて追加されます)。
- **フリーズ対策**: 壊れた/巨大なファイルの解析が固まった場合に全体が止まらないよう、
  1ファイルあたりの抽出処理に60秒のタイムアウトを設けています(タイムアウトした
  ファイルはエラー扱いとして記録し、次のファイルへ進みます)。万一それでも作成処理が
  止まって見える・キャンセルが効かない場合は、アプリを再起動してください(進捗は
  メモリ上の状態のため、再起動すれば必ず解消します。それまでに処理済みだった分は
  DBに保存済みなので失われません)。
- **検索結果の表示**: 各行の「一致の根拠」に、ファイル名一致/本文一致/意味検索(類似度・
  本文の抜粋)がバッジで表示されます。
- **ファイルプレビュー**: 検索結果の「プレビュー」ボタンで、ファイルを開かずに本文の
  先頭(最大4,000文字)を確認できます。コンテンツインデックス済みの本文は即表示され、
  未作成・ファイル更新後の場合はその場で読み取ります(対応形式のみ。読み取り専用で
  索引には影響しません)。
- **大規模データでの高速化(IVF近似検索)**: チャンク数が5,000件を超えると、意味検索は
  自動で近似最近傍探索(クラスタリングによる絞り込み)に切り替わります。インデックスは
  コンテンツインデックス作成の最後に必要な場合のみ自動構築・更新され(進捗欄に
  「意味検索用インデックスを最適化中」と表示)、追加のネイティブ依存はありません。
  近似のため、ごく一部の類似ファイルが結果に入らないことがありますが(キーワード
  検索・本文全文検索は常に全件が対象で影響なし)、走査量が約1/8になり大幅に高速化
  されます。インデックス構築後に追加されたファイルも検索漏れは起きません(未割当の
  チャンクは常に走査対象に含まれます)。
- **既知の制約**: 意味検索の類似度の閾値・件数は`data/config.json`の`phase1`セクション
  (`semantic_min_score` / `semantic_max_results`)で調整できますが、Phase 0/1の試作段階
  では経験的なデフォルト値(0.75 / 10件)のままです。

## 旧バージョンからの更新時の注意

- 差分スキャンの性能問題(検索が長時間固まる)を修正した版へ更新した場合、
  既存の `data/mitsukaru.db` は起動時に自動移行されます。ただし旧版で肥大化した
  DBを整理し、OneDrive配下など旧版で索引から漏れていたファイルを取り込むため、
  **更新後に設定画面から一度「全再スキャン」を実行することを推奨**します
  (よりきれいにしたい場合はアプリ停止中に `data/mitsukaru.db` を削除して再スキャン)。
